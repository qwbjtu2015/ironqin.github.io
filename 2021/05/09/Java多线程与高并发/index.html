Java多线程编程

1. 多线程编程

1.1 线程简介

  1. 程序、进程（Process）和线程（Thread）：一个软件运行是一个进程，而一个进程包含多个线程，而程序是静态的，可执行。
  2. 进程是系统资源分配的基本单位，一个进程中至少有一个线程（Main线程），线程是CPU调度和执行的单位。
  3. 实际很多多线程是模拟而来，是由CPU快速切换线程而实现的。
  4. 对统一资源的控制需要并发控制，防止同时操作

1.2 线程实现 【重要】

  1. 线程的三种创建方式：继承Thread类，实现Runnable接口、实现Callable接口。
  2. 继承Thread类：继承Thread类，然后重写run方法，启动时使用start方法，主线程和该线程会交替执行，同步执行。如果是
  3. 线程开启不一定立即执行，由CPU进行调度。
  4. 实现Runnable方法：创建类，实现Runnable接口，重写run方法，然后创建Thread类，传入我们的类，再调用start方法。
  5. 多个线程同时操作同一个对象：
  6. 实现Callable接口：创建一个类，实现Callable接口，重写call方法（有返回值），然后创建执行服务，提交执行、获取结果，最后关闭服务。
  7. 静态代理模式：
    • 真实对象和代理对象都要实现同一个接口
    • 代理对象要代理真实角色，因此代理对象中要传入真实对象
    • 好处是：代理对象可以做很多真实对象做不了的事情；真实对象可以专注做自己的事情
    • 线程的Runnable接口实现就是静态代理模式，Thread类和我们定义的类都实现了Runnable接口，通过Thread对象的start方法启动了我们的对象。
  8. Lambda表达式
    • 避免内部类定义过多
    • 使代码看起来更简洁
    • 去掉了没有意义的代码，只保留核心逻辑

1.3 线程状态

  1. 五个状态：创建、就绪、运行、阻塞、死亡

    • sleep() 线程休眠
    • join()
    • yield()
    • isAlive()

  1. 线程停止：一般不使用JDK提供的stop()/destroy()方法（已废弃）。推荐线程自己通过标志位进行停止。
  2. sleep休眠：
    • 时间参数指定线程阻塞的毫秒数
    • 会产生异常InterruptedException，需抛出或捕获
    • 休眠时间结束后会进入就绪状态
    • 可以用来模拟网络延迟（放大问题的发生性）和倒计时
    • 每个对象都有一个锁，sleep不会释放锁
  3. yield礼让：
    • 礼让线程，让电气概念正在执行的线程暂停，但不阻塞
    • 将线程从运行状态转为就绪状态
    • 让CPU重新调度，礼让不一定成功，也可能该线程继续被CPU调度
  4. join合并：待该线程执行后，其他线程才能运行，相当于插队
  5. 线程状态观测Thread.State
    • NEW 创建后未启动
    • RUNNABLE 正在运行
    • BLOCKED 阻塞状态
    • WAITING 阻塞
    • TIMED_WAITING 阻塞
    • TERMINATED 结束，死亡
  6. 线程优先级：最小为1，最大为10，默认为5。提高优先级只是提高了CPU先调度的概率，这并不意味着优先级高的一定比优先级低的先被执行。优先级的设定建议在start()函数之前。
  7. 守护线程daemon：
    • 线程分为用户线程和守护线程
    • 虚拟机必须确保用户线程执行完毕
    • 虚拟机不用等待守护线程执行完毕
    • 守护线程：如后台记录操作日志、垃圾回收（GC）、内存监控等 虽然God线程是一个死循环，但是程序仍然会正常退出，这是因为它是守护线程，JVM并不会等待它完成，JVM只会等用户线程完成。

1.4 线程同步 【重要】

  1. 并发：同一个对象被多个线程同时操作。线程同步实际上就是一种等待机制，多个需要同时访问的线程进入这个对象的等待池形成队列，等待前面线程使用完毕，下一个线程再使用。除了有队列之外，还需要锁，才能保证安全。
  2. 队列和锁：
    • 锁机制：synchronized:当一个线程获得对象的排它锁，独占资源，其他线程必须等待，使用后释放锁即可，存在以下问题：
      • 一个线程持有锁会导致其他所有需要这个锁的线程被挂起
      • 多线程下频繁的加锁和释放锁操作会导致大量的上下文切换和调度延时，引发性能问题
      • 如果优先级高的线程等待优先级低的线程释放锁，会导致优先级倒置，引发性能问题
  3. synchronized关键字：synchronized方法或synchronized块
    • 同步方法必须获得调用该方法的对象的锁才能执行，否则线程会阻塞
    • 方法一旦执行，就会独占该锁，知道该方法返回才释放
    • 缺陷：若将一个大的方法申明为synchronized将会影响效率。
    • 同步方法直接是 java public synchorized void buy(){...} 对于方法锁住的默认是this，也就是对象。
    • 同步块： java synchorized(Obj) {}
    • Obj称为同步监视器
    • Obj可以是任何对象但是一般推荐共享资源作为同步监视器
    • 同步方法中无需指定同步监视器，因为同步方法的同步监视器就是this，也就是对象本身，或者是class
    • 同步监视器的执行
      • 第一个线程访问，锁定同步监视器，执行代码块
      • 第二个线程访问，发现同步监视器被锁定，无法访问
      • 第一个线程访问完之后，解锁同步监视器，第二个线程可以访问了
  4. 死锁产生的四个必要条件
    • 互斥：一个资源每次只能被一个线程使用
    • 请求与保持：一个线程请求资源而被阻塞时，对已获得的资源保持不放
    • 不可剥夺：线程已获得的资源，在未使用完之前，不能强制剥夺
    • 循环等待：若干线程之间形成一种头尾相接的循环等待关系
      只需破坏其中任意一个条件，就可以避免死锁。
  5. Lock(锁) 从JDK5开始，显示定义同步锁对象来实现同步，同步锁使用Lock对象充当。Lock接口是控制多个线程对共享资源进行访问的工具。
  6. ReentrantLock类（可重入锁）实现了Lock接口，拥有与synchronized相同的并发性和内存语义，可以显示加锁和释放锁。
  7. synchronized和Lock对比
    • Lock是显式锁，手动加锁和释放，synchronized锁是隐士锁，出作用域后自动释放
    • Lock只有代码块锁，synchronized还有方法锁
    • 使用Lock锁，JVM会花费较少的时间来调度线程，性能更好，并且有更好的扩展性
    • 优先使用顺序：Lock > 同步代码块 > 同步方法

1.5 线程通信问题

  1. 线程协作（生产者消费者问题）Object类中的两个方法：wait()和notify()，只能在同步方法或同步代码块中使用。

    • wait：表示线程一直等待，直到其它线程通知，与sleep不同，它会释放锁
    • notify：唤醒一个处于等待状态的线程
    • notifyAll()：唤醒同一个对象上所有调用wait()方法的线程，优先级高的线程优先调度

  2. 生产者消费者解决方案

    • 管程法（生产者->缓冲区->消费者）
      • 生产者：负责生产数据的模块（方法/对象/线程/进程）
      • 消费者：负责处理数据的模块
      • 缓冲区：消费者不能直接使用生产者的数据，中间存在缓冲区
    • 信号灯法（信号量法）相当于缓冲区大小为1的管程法，使用布尔flag进行管理。

  3. 线程池：提前创建好多个线程，放入线程池，使用时直接获取，使用完放回池中，可以避免频繁创建销毁，实现重复利用。
    优点：

    • 提高响应速度（节省了线程创建时间）
    • 降低资源消耗（重复利用线程）
    • 便于线程管理
      • corePoolSize:核心池的大小
      • maximumPoolSize：最大线程数
      • keepAliveTime：线程没有任务时最长保持多久终止

  4. 使用线程池：ExecutorService：线程池接口，常见子类为ThreadPoolExecutor。
