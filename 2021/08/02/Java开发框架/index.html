介绍常见的Java开发框架及知识要点

Spring/Spring Framework/Spring Boot/Spring Cloud

基本概念区分

Spring是一个生态体系，包含了Spring Framework、Spring Boot、Spring Cloud等等。

Spring Framework：它是整个Spring生态的基石，是一个轻量级java开发框架，主要为依赖注入、事务管理、Web应用、数据访问提供核心支持。

Spring Boot：Spring Framework虽然接口简化，设计优美，但是它本身不是容器，所以必须依赖于Tomcat等容器，而Spring Boot更简单，集成度更高，信仰约定大于配置，不需要原来繁杂的xml配置，使web应用开发大大简化。

Spring Cloud：它是一整套基于Spring Boot的微服务解决方案，提供了很多组件，如配置管理、注册中心、服务发现、限流、网关、链路追踪等等，它的每一个组件都是基于Spring Boot构建的。但是这些组件很多不是非常成熟，因此各大公司会开发自己的组件，所以说Spring Cloud的更多的是提供了一种设计理念，并不局限于某些组件。

Tomcat与War包：Tomcat是一个轻量级web应用服务器，在中小型系统和并发访问用户较少的场景下使用。War包是在进行web开发时一个Project下的所有源码的集合，既包含前端HTML、CSS、JS代码，也包含后台Java逻辑。war包生成后可以直接放在tomcat下面的webapps目录下，随着tomcat启动，它可以被自动解压。

Spring Boot多模块应用和Spring Cloud微服务应用的区别

Spring Boot的单模块和多模块：Spring的多模块指的是一个Project包含多个Module，但是只有最外层的Project被build为Jar包，里面的Module不会被单独打包，而且这些module都是被运行在同一个JVM里面。而Spring Cloud中的组件是可以独立打包成为Jar包的，而且他们可以运行在不同的JVM里，如果使用容器部署，他们在不同的docker容器中，也就是有可能运行在不同的物理机上，这就是微服务，组件之间的调用是跨JVM的，一般通过http通信。微服务的优点是错误隔离、容易监控、定点扩缩容。占用资源的服务被单独拿出来，可以分配更多的资源，满足其需求。微服务的拆分要参考很多因素（技术、组织、产品等等），总之原则是架构清晰、开发方便、部署简单、维护容易。

Spring Bean

Bean定义

在Spring中，哪些组成应用程序的主题以及由Spring IoC容器管理的对象，被称为Bean。Bean的定义以及Bean之间的依赖关系将通过配置元数据来描述。

Bean的作用域

  • singleton：唯一Bean实例，Spring中的bean默认都是单例的
  • prototype：每次请求都会创建一个新的bean实例
  • request：每一次HTTP请求都会产生一个新的bean，该bean仅在当前HTTP request内有效
  • session：每一次HTTP请求都会产生一个新的bean，该bean仅在当前HTTP session内有效
  • global-session：全局session作用域，Spring5中已取消

Spring中的单例bean线程安全问题

单例bean全局只有一个实例，当多个线程操作同一个对象的时候，对该对象的成员变量的写操作会存在线程安全问题。但是我们常用的Controller、Service和Dao这些都是无状态的，无状态的bean不能保存数据，所以是线程安全的。如果有需要修改变量的类，我们可以在类中定义一个ThreadLocal成员变量，将需要修改的成员变量保存在ThreadLocal中（推荐方式）。第二种方法，我们可以把这个Bean的作用域改为prototype，每次请求创建一个新实例，自然没有线程安全问题。

@RestController 和 @Controller

Controller返回一个页面，RestController返回JSON或XML形式数据。简单来讲，Controller返回一个View视图，而RestController只需要返回一个对象

Spring IOC & AOP

Spring框架的核心技术：依赖注入和面向切面编程

  • 依赖注入(DI)/控制反转(IoC)

依赖注入和控制反转描述的是同一件事情，那就是类和类的依赖关系，比如A类要调用B类，那么就要在A类中创建B类的对象，然后进行调用，这样在大规模项目中会使得程序耦合过于紧密，如果要替换这个类，就需要修改所有调用了这个类的代码。IoC和DI就是将这个类的依赖关系解耦，把它们放到Spring框架中去做。那么本来由类控制其所依赖的类的对象的创建，变成了框架控制类对象的创建，这就叫“控制反转”。而从类的使用角度看，是框架创建了对象之后，将这个对象注入到了依赖它的所有的类中，这就是“依赖注入”。

  • 面向切面编程(AOP)

Spring MVC工作原理

Fig2

  • 客户端发送请求，直接请求到DispatcherServlet（前端控制器）
  • DispatcherServlet根据请求信息调用HandlerMapping，解析请求对应的Handler
  • 解析到Handler也是我们所说的Controller后，开始由HandlerAdapter适配器处理
  • HandlerAdapter会根据Handler来调用真正的处理器来处理请求，并处理相应的业务逻辑
  • 处理器处理完成后会返回一个ModelAndView对象，Model是返回的数据对象，View是个逻辑上的View
  • ViewResolver会根据逻辑View查找实际的View
  • DispatcherServlet把返回的Model传给View进行视图渲染
  • 把View返回给浏览器

事务的隔离级别（5个）：读未提交、读已提交、可重复读、串行化 、使用后端数据库默认隔离级别

事务传播行为：

  • 支持当前事务
    • 如果当前存在事务，则加入该事务；如果当前没有事务，则创建一个新的事务【默认】
    • 如果当前存在事务，则加入该事务；如果当前没有事务，则以非事务的方式继续运行
    • 如果当前存在事务，则加入该事务；如果当前没有事务，则抛出异常
  • 不支持当前事务
    • 创建一个新的事务，如果当前存在事务，则把当前事务挂起
    • 以非事务方式运行，如果当前存在事务，则把当前事务挂起
    • 以非事务方式运行，如果当前存在事务，则抛出异常
